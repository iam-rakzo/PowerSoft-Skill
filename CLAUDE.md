# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A Claude Code plugin (`powersoft`) that ships two workflow skills for Symfony/PHP projects. No build step, no runtime — all artifacts are Markdown and JSON.

## Plugin structure

```
.claude-plugin/
  plugin.json        # Plugin identity (name, version, author)
  marketplace.json   # Marketplace listing metadata
skills/
  <skill-name>/
    SKILL.md         # Skill definition: YAML frontmatter + Markdown body
```

Each `SKILL.md` must have frontmatter with `name` and `description`. The `description` field is what Claude Code reads to decide when to invoke the skill.

## Skills included

| Skill | Purpose |
|-------|---------|
| `start-work` | Verify plan → sync `develop` → create typed branch |
| `finish-work` | Run validations → draft PR body → `gh pr create` |

### Branch convention (enforced by `start-work`)

Format: `<type>/<YYYY-MM-DD>-<slug>`  
Example: `feat/2026-04-24-agregar-strava-webhook`

Types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `build`, `perf`, `revert`, `ci`

### Validations enforced by `finish-work` (all mandatory, in order)

1. `php bin/console lint:container`
2. `php bin/phpunit`
3. `npx playwright test` (skipped with manual confirmation if not configured)
4. Manual browser verification

PRs are always opened against `develop`, not `main`.

## Adding a new skill

1. Create `skills/<skill-name>/SKILL.md` with frontmatter:
   ```markdown
   ---
   name: skill-name
   description: When Claude Code should invoke this skill...
   ---
   ```
2. Register the skill in `plugin.json` if needed (currently auto-discovered from `skills/`).

## Installing locally for testing

```bash
claude /install-plugin github:oscrodriguez/powersoft
```

Or from a local path during development — point the install command at the repo directory.

## Key constraints in skill design

- `start-work` is **gated**: blocks if no approved plan exists in `docs/superpowers/plans/`.
- `finish-work` is **gated**: blocks on any failed validation — no skipping allowed.
- Branch names are normalized (lowercase, hyphens, no accents/special chars, max 40 chars for the slug).
- The linked plan path is carried from `start-work` into the PR body generated by `finish-work`.
